# Module 6: Concurrency - Web Server

## Reflections

### Milestone 1: Listening to TCP Connections
In this milestone, I learned how to establish a basic TCP listener using Rust's `std::net::TcpListener`. The `bind` function allows the server to listen on a specific IP address and port, which in this case was `127.0.0.1:7878`. The `incoming()` method returns an iterator over the connection attempts, and each successful connection provides a `TcpStream`. This stream represents the connection between the client and the server, allowing for data exchange. Understanding how to unwrap these streams and handle potential errors is crucial for building a robust network application. This foundational step set the stage for processing HTTP requests in subsequent milestones.

### Milestone 2: Returning HTML
Milestone 2 focused on transforming the server from a simple connection logger into a functional web server that returns HTML content. I implemented the `handle_connection` function, which reads the HTTP request and sends back a hardcoded HTTP response. By using `fs::read_to_string`, I could serve the contents of `hello.html` directly to the client's browser. The response followed the standard HTTP format, including the status line, headers like `Content-Length`, and the message body. This exercise highlighted the importance of following protocol standards to ensure the browser correctly interprets and renders the server's output. It was satisfying to see a real web page being served from my own Rust application.

### Milestone 3: Validating Requests
In this milestone, I improved the server's logic to distinguish between valid and invalid request paths. By reading the first line of the HTTP request, known as the request line, I could check if the client was asking for the root path (`/`) or something else. I refactored the code to use an `if/else` block (and later a `match` statement) to selectively choose the appropriate response status and HTML file. If a request was not for the root, the server would return a `404 NOT FOUND` status along with a custom `404.html` page. This refactoring reduced code duplication and made the server more professional by handling non-existent routes gracefully. It also demonstrated how to manage multiple resources and response states effectively.

### Milestone 4: Simulation of Slow Response
Milestone 4 introduced a simulation of a slow request using `thread::sleep` to understand the limitations of a single-threaded server. By adding a `/sleep` route that delays the response by 10 seconds, I observed how this single request blocked all other incoming connections. This meant that while the server was "sleeping" for one client, no other client could get a response, even for the standard `/` path. This phenomenon is known as "head-of-line blocking" and perfectly illustrated the need for concurrency. It was a clear demonstration that a production-grade server must be able to handle multiple tasks simultaneously to remain responsive to all users. This realization was the driving force behind moving to a multithreaded architecture in the final milestone.

### Milestone 5: Multithreaded Server
In the final milestone, I implemented a `ThreadPool` to handle multiple requests concurrently, significantly improving the server's throughput. Instead of processing each connection sequentially, the server now dispatches jobs to a pool of worker threads that execute the `handle_connection` function in parallel. I used a channel (`std::sync::mpsc`) to send tasks from the main thread to the workers, ensuring safe communication. The worker threads were wrapped in an `Arc<Mutex<Receiver>>` to allow multiple threads to safely access the shared receiver. This architecture prevents a single slow request, like `/sleep`, from blocking the entire server, as other workers remain available to handle new connections. Implementing graceful shutdown by using the `Drop` trait for the `ThreadPool` ensured that all threads are properly joined before the program terminates.

### Bonus: build vs new
For the bonus task, I implemented a `build` function as an alternative to the `new` constructor for the `ThreadPool`. While `new` uses `assert!` to panic if the requested pool size is zero, `build` returns a `Result<ThreadPool, PoolCreationError>`, allowing the caller to handle the error more gracefully. This approach aligns with Rust's philosophy of explicit error handling rather than crashing the program. Comparing the two, `build` is generally preferred in production environments where you might want to log the error or fall back to a default configuration. It provides more flexibility and safety, as it forces the developer to acknowledge and handle the possibility of a failed initialization. 

In my implementation in `main.rs`, I utilized `unwrap_or_else` on the `Result` returned by `build`. This allowed the program to print a helpful error message and exit cleanly with a status code of `1`, rather than panicking with a stack trace. This functional improvement makes the application more robust and user-friendly. By avoiding panics, I can ensure that any necessary cleanup (though minimal in this specific case) could potentially be performed before the process terminates. This exercise reinforced my understanding of when to use panics versus when to use `Result` for better API design and overall application reliability.
